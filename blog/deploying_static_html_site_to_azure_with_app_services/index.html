Deploying a Static HTML Site to Azure with App Services

October 18, 2018 Â· 2 minute read

What’s a Static HTML Site?

… you may be asking yourself.

It’s like the hip way to be old school, but in a new-skool kinda way!

Nowadays, most websites contain both server-side and client-side (browser) scripts that are actively changing the content and properties of the website. Any e-commerce site is a good example of this.

Data is stored and delivered to the server and browser from a variety of sources and served up to the end-user dynamically as they navigate their way through the site. And that’s just browsing the site. Once you start talking about transacting with things like payment systems or third-party services you could safely say that the website is definitely NOT static.

Content Management Systems (CMS) such as Wordpress, Joomla, and Drupal are often used for basic blogs and personal sites. Administration of rogue “plugins” not playing well with other aspects of the site, they are often burdonsome and overkill when you just need a basic website.

A site that doesn’t have content that is constantly updating.

One that isn’t “dynamic” as they say.

That’s the use case for a static HTML site.

Examples (JasonHand.com, CommunityPulse.io, DevOpsDays.org)

This website is a static HTML website.

The only thing that changes is an occasional new blog post that includes a couple of new images.

The overhead on managing a website in this way is super low!

Huzzah! I feel like a modern Web Master.

Using an Azure App Service, we can deploy an HTML website in seconds.

Not only that but because we’ll configure it to look at our GitHub repository (where the HTML is store), any time I push new code to my master branch, the App Service will automatically deploy my site to reflect the changes.

So, any time I have a new blog I’m ready to post, all I have to do is push my code up to GitHub and go pour myself a cold one.

Oh.. and it wont’ cost me anything. Yep. Nada.

This is the process I use for myself. It’s the process my co-hosts at Community Pulse use. It’s the process that DevOpsDays Rockies (and the rest of the DoD’s) use.

Old-school basic HTML is back in style, baby!
